1. Project Overview

CampusLibra is a comprehensive library management system designed specifically for educational institutions. The system provides a complete solution for managing library operations including book cataloging, user management, borrowing and return processes, reservation queue management, and fine calculation.

Key Objectives

  • Streamline library operations with digital automation
  • Provide role-based access for different user types (Admin, Librarian, Member)
  • Enable efficient book search and catalog management
  • Track borrowing activities and manage due dates
  • Implement fair reservation queue system
  • Automate fine calculation for overdue books
  • Provide real-time notifications for important events

Core Features

✓ User Authentication & Authorization
✓ Book Catalog Management
✓ Borrowing & Return System
✓ Reservation Queue Management
✓ Fine Calculation & Payment
✓ Real-time Notifications
✓ Audit Trail System
✓ Dashboard & Analytics

Project Goals

The primary goal of CampusLibra is to modernize library operations by replacing manual processes with an efficient, user-friendly digital system. The application ensures data integrity, provides real-time updates, and offers comprehensive reporting capabilities for library administrators.

3. System Architecture

Architecture Overview

CampusLibra follows a three-tier architecture pattern consisting of:

  1. Presentation Layer (Frontend): Angular application providing user interface
  2. Application Layer (Backend): Express.js REST API handling business logic
  3. Data Layer (Database): MongoDB storing application data

User Roles & Permissions

Role Access Level Key Permissions
Admin Full System Access All CRUD operations, user management, system settings, role assignment
Librarian Operational Access Book management, process borrows/returns, manage reservations, view users
Member Limited Access Browse books, borrow/return, create reservations, view/pay fines

Main System Entities

User Management

  • Users
  • Roles
  • Permissions

Book Management

  • Books
  • Book Copies
  • Book Audits

Circulation

  • Borrows
  • Reservations
  • Fines

System

  • Settings
  • Notifications
  • Dashboard Stats

Data Flow

  1. User interacts with Angular frontend
  2. Frontend makes HTTP request to Express API
  3. API validates request and checks authentication/authorization
  4. Business logic is executed in service layer
  5. Data is retrieved/modified in MongoDB database
  6. Response is sent back through the chain to user
  7. Real-time updates are pushed via Socket.IO when applicable

6. Database Design

Database Models

User Model

{ name: String (required), email: String (required, unique, lowercase), password: String (required, hashed, select: false), role: ObjectId → Role, isBlocked: Boolean (default: false), createdAt: Date, updatedAt: Date }

Book Model

{ title: String (required), author: String (required), isbn: String (unique, sparse), publishedDate: Date, category: String, createdAt: Date, updatedAt: Date, copies: Virtual → BookCopy[] }

BookCopy Model

{ book: ObjectId → Book (required), Barcode: String (unique, sparse), status: String (default: 'available'), location: String, createdAt: Date, updatedAt: Date }

Borrow Model

{ user: ObjectId → User (required), book: ObjectId → Book (required), bookCopy: ObjectId → BookCopy (required), borrowDate: Date (required), dueDate: Date (required), returnDate: Date, status: String (enum: ['active', 'returned', 'overdue']), renewalCount: Number (default: 0), createdAt: Date, updatedAt: Date }

Reservation Model

{ user: ObjectId → User (required), book: ObjectId → Book (required), status: String (enum: ['pending', 'ready', 'fulfilled', 'cancelled']), queuePosition: Number, reservationDate: Date (default: now), holdExpiresAt: Date, createdAt: Date, updatedAt: Date }

Fine Model

{ user: ObjectId → User (required), borrow: ObjectId → Borrow (required), amount: Number (required), reason: String, isPaid: Boolean (default: false), paidAt: Date, createdAt: Date, updatedAt: Date }

Role Model

{ name: String (required, unique, enum: ['admin', 'librarian', 'member']), description: String (required), permissions: [ObjectId] → Permission[], createdAt: Date, updatedAt: Date }

Permission Model

{ name: String (required, unique), description: String (required), category: String (enum: ['book', 'user', 'borrow', 'reservation', 'fine', 'role', 'system']), createdAt: Date, updatedAt: Date }

Notification Model

{ recipient: ObjectId → User (required), type: String (required), title: String (required), message: String (required), isRead: Boolean (default: false), relatedId: ObjectId, relatedType: String, createdAt: Date, updatedAt: Date }

Setting Model

{ key: String (required, unique), value: Mixed (required), description: String, createdAt: Date, updatedAt: Date }

BookAudit Model

{ book: ObjectId → Book (required), bookCopy: ObjectId → BookCopy, action: String (required), performedBy: ObjectId → User (required), details: Mixed, timestamp: Date (default: now), createdAt: Date, updatedAt: Date }

Database Indexes

  • Users: email (unique), role
  • Books: title & author (text search), category, isbn (unique)
  • BookCopies: book, status, Barcode (unique)
  • Borrows: user, book, status, dueDate
  • Reservations: user, book, status
  • Fines: user, isPaid

10. Challenges & Solutions

Challenge 1: CORS Configuration for Production

Problem: Frontend deployed on Netlify could not communicate with backend on Render due to CORS policy restrictions.

Solution: Implemented dynamic CORS configuration in Express.js that accepts multiple origins:

const allowedOrigins = [ 'http://localhost:4200', // Development process.env.FRONTEND_URL // Production ].filter(Boolean); app.use(cors({ origin: (origin, callback) => { if (!origin || allowedOrigins.includes(origin)) { callback(null, true); } else { callback(new Error('Not allowed by CORS')); } }, credentials: true }));

Challenge 2: Real-time Notification System

Problem: Needed to notify users instantly about important events (returns, reservations, blocks) without polling.

Solution: Integrated Socket.IO for WebSocket-based real-time communication. Server emits events, and Angular service listens for updates and displays notifications.

Challenge 3: Role-Based Permission System

Problem: Different user roles needed granular permissions for different operations.

Solution: Implemented a flexible permission system with Permission model and Role model having many-to-many relationship. Created middleware to check both role and specific permissions.
